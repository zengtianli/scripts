# 高水位方案需水量问题分析 & 网页端改造方案

## 一、问题描述

钱管局反馈：高水位方案应该需水更多，但计算结果显示需水更少。

## 二、问题分析

### 2.1 当前计算逻辑

```
需水量 = 农业需水 + 非农需水          ← 来自 input 文件，与水位无关
总需水量 = 需水量 + 生态需水
缺水(浙东需供) = max(总需水量 - 合计来水, 0)   ← 也与水位无关
```

关键发现：**当前代码中，需水量和缺水(浙东需供)的计算完全不依赖初始水位**。

两个方案的区别仅在 `input_SW_CS.txt`（初始水位），但这只影响：
- 日初容积（高水位 → 更大的初始蓄水）
- 排水量（蓄水多 → 可能触发更多排水）
- 排末容积

**不影响**：需水量、总需水量、缺水(浙东需供)。

### 2.2 可能的问题根源

#### 假设 A：「需水」指的是「缺水(浙东需供)」

如果钱管局说的「需水更多」指的是浙东需要供给更多水：

- 当前逻辑：`缺水 = max(总需水量 - 合计来水, 0)`
- 这个公式与水位无关，所以两个方案的缺水量应该**完全相同**
- 如果结果不同，说明两个方案的 input 文件本身就不同（不仅仅是 SW_CS）

**需要确认**：高水位方案和当前水位方案，除了 `input_SW_CS.txt` 之外，还有哪些 input 文件不同？

#### 假设 B：高水位方案应该考虑「维持高水位的额外需水」

钱管局的逻辑可能是：
- 高水位方案 → 目标是维持高水位 → 需要更多水来补充蒸发/渗漏/排水损失
- 当前代码没有实现这个逻辑

如果是这个意思，需要新增计算：
```
维持水位需水 = 目标容积 - 日中容积（当日中容积 < 目标容积时）
```

#### 假设 C：生态需水应该与水位挂钩

代码中 `水位生态需水` 被简化为 0：
```python
# main.py:287
balance_df["水位生态需水"] = 0.0  # 简化后设为0
```

如果高水位方案意味着更高的生态水位要求，那么水位生态需水应该更大，从而总需水量更大。

### 2.3 待确认问题

1. **两个方案的差异**：高水位方案和当前水位方案，具体哪些 input 文件不同？只是 `input_SW_CS.txt` 不同，还是 `input_SW_MB.txt`（目标水位）也不同？
2. **「需水更多」的含义**：指的是「总需水量」还是「缺水(浙东需供)」？
3. **是否需要新增逻辑**：是否需要加入「维持目标水位的额外需水」计算？
4. **水位生态需水**：是否需要恢复水位生态需水的计算（目前被简化为 0）？

## 三、网页端改造方案

### 3.1 现状

当前 `index.html` 只是渲染 `README.md` 的静态页面，没有展示任何计算数据。

### 3.2 改造目标

在网页端展示各河区的需水汇总数据，让用户能直观看到：
- 每个河区的总需水量、缺水量、合计来水等关键指标
- 全部河区的汇总数据
- 方便对比不同方案的结果差异

### 3.3 方案选项

#### 方案 A：纯前端读取 output 文件（推荐）

直接用 JS 读取已有的 `output_hq_*.txt` 和 `output_hq_all_total.txt`，在页面上渲染表格。

优点：
- 不需要后端，不需要 Python 运行环境
- 计算完成后直接打开 index.html 就能看结果
- 改动最小

实现思路：
```
index.html
├── 读取 output_hq_all_total.txt → 展示全河区汇总
├── 读取各 output_hq_*.txt → 展示各河区逐日数据
├── 读取 output_hq_all.txt → 展示逐日汇总
└── 可切换查看不同河区
```

展示内容：
- 汇总卡片：总需水量、总合计来水、总缺水(浙东需供)、总河区排水
- 河区对比表：每个河区一行，列出关键指标的合计值
- 逐日趋势图（可选）：用简单的图表库展示需水/来水趋势

#### 方案 B：Streamlit 应用

用 Streamlit 做交互式仪表盘。

优点：
- 交互性强，可以做筛选、对比
- 可以直接调用 main.py 重新计算

缺点：
- 需要 Python 环境运行
- 与现有 index.html 方案不一致

#### 方案 C：增强现有 index.html + 生成 JSON

main.py 计算完成后额外输出一个 `output_summary.json`，前端读取 JSON 渲染。

优点：
- 数据格式更友好
- 前端解析更简单

缺点：
- 需要改 main.py 增加 JSON 输出

### 3.4 建议

推荐方案 A，改动最小，效果直接。前端直接解析 TSV 文件即可，不需要改后端。
